## Findings / caveats

- **kv8cast-step500 has the best RULER of any model here (0.814)** — driven largely by `niah_multikey_2` = 0.80 vs 0.24 for kv16-step500 (that task is the main differentiator; most others are saturated at 1.0). Its GPQA (0.520) is the lowest of the QAD variants, so it trades reasoning for retrieval.
- **kv16-step500**: GPQA 0.535, RULER 0.773 — middle of the pack.
- **`ruler_vt` = 0.000 on both** (as in 001/002, every model) — systematic non-thinking task/format artifact, not a quant effect. Kept in the 13-task mean.
- All 28 jobs reached intended totals with 0 errored trials (verified against the eval service).

## Ops notes

- **vLLM internal-port collision:** launching 4 servers concurrently, one (port 8002) crashed at startup with `torch.distributed ... EADDRINUSE port 46841` (the API `--port` being distinct is not enough). Fixed by relaunching with a distinct `VLLM_PORT`. Set a unique `VLLM_PORT` per server when running a fleet.
- **Eval-service submit rate limit = 20 jobs/minute:** submitting all 28 jobs at once got 7 kv8cast RULER tasks rejected with HTTP 429 (`submit rate limit exceeded for minute window`). They were resubmitted in a smaller batch and completed. Batch submissions under 20/min going forward.
